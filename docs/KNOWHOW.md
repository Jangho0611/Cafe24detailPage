# Project Know-how

## 프로젝트 철학

이 프로젝트는 카페24 상세페이지 제작을 자동화하되, AI가 임의로 광고 문구나 제품 정보를 만들어내지 않도록 통제하는 것을 핵심으로 한다.

목표는 화려한 결과물이 아니라, B2B 구매자가 제품을 빠르게 이해하고 확인할 수 있는 신뢰도 높은 상세페이지와 보조 인포그래픽을 만드는 것이다.

## 설계 원칙

- 제품군 DB를 기준으로 분류, 구조, 용도, 인포그래픽 표현을 일관되게 유지한다.
- 입력값과 DB 기준이 충돌하면 DB 기준을 우선한다.
- AI는 구조를 창작하는 주체가 아니라, 정해진 기준을 시각적으로 정리하는 도구로 사용한다.
- 제품 설명, HTML, 인포그래픽은 서로 역할이 다르며 같은 내용을 반복하지 않는다.
- 실제 확인된 정보가 부족하면 섹션을 억지로 채우지 않는다.

## A/B/C 타입 역할

### A타입: 비교 전용

A타입은 비교 대상 간 차이를 보여주는 이미지다.

- 1단: 비교 대상
- 2단: 핵심 차이 비교
- 3단: 구조 차이가 있는 경우에만 좌우 구조 비교

구조 차이가 없는 비교에서는 3단을 생성하지 않는다. 예를 들어 일반합판 원산지 비교처럼 내부 구조가 동일한 경우에는 2단에서 종료한다.

### B타입: 구조 설명 전용

B타입은 단일 제품의 구조를 설명하는 이미지다.

- 단면 구조
- 구조 상세 확대
- 핵심 구조 키워드

A타입에서 B타입의 단일 제품 구조 설명을 복사하지 않는다.

### C타입: 표면 질감 설명 전용

C타입은 표면 상태나 질감 차이를 보여주는 이미지다.

- 표면 질감
- 표면 비교
- 핵심 질감 키워드

## Human Writing Rule

문장은 사람이 현장에서 쓰는 짧고 중립적인 표현을 기준으로 한다.

- 광고 문구 금지
- 과장 표현 금지
- 근거 없는 추천 금지
- 제품명 반복 금지
- 구매자가 실제로 확인할 수 있는 표현 우선

좋은 표현은 “표면 상태 확인”, “재단 치수 확인”, “마감 조건 확인”처럼 짧고 구체적이다.

## Product Fidelity 원칙

Product Fidelity는 제품의 실제 구조와 속성을 유지하는 원칙이다.

- 합판은 단판 적층 구조로만 표현한다.
- MDF는 목재 섬유 압축 구조로만 표현한다.
- PF보드는 PF 폼 코어와 면재 구조로 표현한다.
- XPS는 폐쇄 셀 압출 발포 구조로 표현한다.
- 데크재는 목재 결, 측면 단면, 피스 또는 클립 체결 구조 중심으로 표현한다.

다른 제품군의 구조를 섞으면 실패로 본다.

## Evidence Validation 원칙

근거가 없는 정보는 만들지 않는다.

- 수치
- 인증
- 등급
- 내구성
- 수명
- 강도
- 가격 우위
- 품질 우열

위 정보는 입력 데이터, 제조사 자료, 제품군 DB, 운영 기준에 명확한 근거가 있을 때만 사용한다.

## Realism 원칙

이미지는 실제 제품 또는 실제 시공 구조처럼 보여야 한다.

- 비현실적인 부품 생성 금지
- 장난감 같은 구조 금지
- 실제 판매되지 않는 연결구 생성 금지
- 구조가 섞인 하이브리드 제품 생성 금지
- 데크재 체결 방식은 실제 가능한 위치와 비율로만 표현

## Graphic Quality 원칙

그래픽은 정보 전달을 돕는 수준에서 정돈되어야 한다.

- B2B 자료 스타일 유지
- 색상 절제
- 텍스트는 짧고 선명하게
- 섹션은 정보 우선순위에 맞게 구성
- 장식보다 가독성 우선
- 빈 정보를 억지로 카드로 채우지 않음

## One Source of Truth

제품군 DB가 단일 기준점이다.

- 제품군 분류
- 구조 표현
- 인포그래픽 구조
- 금지 키워드
- 주요 확인 항목

비교 대상이나 모델의 일반 지식보다 제품군 DB와 운영 기준을 우선한다.

## HTML과 인포그래픽 역할 분리

HTML은 상세 설명과 구매 전 확인 내용을 담당한다.

인포그래픽은 HTML을 반복하지 않고, 눈으로 빠르게 이해해야 하는 정보만 담당한다.

- HTML 문장 반복 금지
- 구매 체크포인트 반복 금지
- 동일한 용도 설명 반복 금지
- 이미지에는 비교, 구조, 질감처럼 시각화 가치가 있는 정보만 배치

## Prompt 작성 원칙

프롬프트는 AI에게 창작 자유를 주는 문장이 아니라, 기준과 금지사항을 명확히 전달하는 운영 문서처럼 작성한다.

- 비교 방향은 명시한다.
- 좌우 라벨은 코드에서 확정한다.
- 제품군별 금지 구조를 명시한다.
- 근거 없는 수치와 인증 생성을 금지한다.
- 구조 차이가 없으면 3단을 만들지 말라고 명시한다.

## 최소 수정 원칙

문제가 발생하면 전체 구조를 바꾸기보다 원인을 만드는 최소 지점만 수정한다.

- 함수 범위를 좁힌다.
- 다른 제품군에 영향이 없는 조건을 둔다.
- 기존 안정 동작은 유지한다.
- HTML/CSS/Schema 변경은 별도 승인 없이는 하지 않는다.

## Sprint 운영 방식

Sprint는 “조사 → 승인 → 최소 수정 → Apps Script Push → 실제 테스트 → 기록” 순서로 진행한다.

- 조사 단계에서는 코드 수정하지 않는다.
- 승인 후에만 수정한다.
- Push, Commit, Git Push는 사용자가 요청할 때만 수행한다.
- context.md는 작업 이력 원본으로 유지한다.
- 정리 문서는 별도 md 파일로 생성한다.
